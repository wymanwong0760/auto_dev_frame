# 长时间运行智能体的高效 Harness 设计

来源：https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents
抓取时间：2026-03-16

---

随着 AI 智能体能力不断增强，开发者越来越希望它们承担复杂任务，而这类任务往往会持续数小时，甚至数天。然而，如何让智能体在跨越多个上下文窗口时持续、稳定地推进工作，仍然是一个尚未完全解决的问题。

长时间运行智能体的核心挑战在于：它们必须以离散会话的形式工作，而每个新会话开始时，都不记得之前发生了什么。可以把这想象成一个软件项目由轮班工程师共同完成，但每位新上班的工程师都完全不知道上一班做了什么。由于上下文窗口有限，而大多数复杂项目又无法在单个窗口内完成，因此智能体需要一种方式，在不同编码会话之间搭起桥梁。

我们设计了一个双重方案，让 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) 能够在多个上下文窗口中高效工作：第一部分是一个初始化智能体，用于在首次运行时搭建环境；第二部分是一个编码智能体，负责在每次会话中做出增量式进展，并为下一次会话留下清晰的工作痕迹。你可以在配套的 [quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding) 中找到代码示例。

## 长时间运行智能体的问题

Claude Agent SDK 是一个强大且通用的智能体 harness，擅长编程，也适用于其他需要模型借助工具来收集上下文、制定计划并执行的任务。它具备上下文管理能力，例如 compaction，这让智能体可以在不耗尽上下文窗口的情况下持续处理任务。从理论上讲，在这样的配置下，智能体应该能够无限期地持续做出有用工作。

然而，仅有 compaction 还不够。开箱即用时，即便让像 Opus 4.5 这样的一流编程模型，在 Claude Agent SDK 中循环跨多个上下文窗口运行，如果只给它一个高层级提示，例如“构建一个 [claude.ai](http://claude.ai/redirect/website.v1.19d5acde-906d-4607-a61a-3fef909d14e3) 的克隆版”，它依然难以产出一个达到生产质量的 Web 应用。

Claude 的失败主要表现为两种模式。第一，智能体往往会试图一次性做太多事情——本质上是想“一把梭”把整个应用做完。这通常会导致模型在实现过程中耗尽上下文，使下一次会话接手时面对的是一个功能只做了一半、而且没有文档说明的状态。此时新的智能体只能猜测之前发生了什么，并花费大量时间重新让基础应用恢复可用。即便有 compaction，这种情况依然会发生，因为 compaction 并不总能把足够清晰的指令传递给下一个智能体。

第二种失败模式通常发生在项目后期。当某些功能已经完成后，后续某个智能体实例会查看当前状态，发现“已经有进展了”，于是直接判断任务已经完成。

这就把问题拆成了两个部分。首先，我们需要建立一个初始环境，为提示中要求的所有功能打下基础，使智能体能够按步骤、按功能逐一推进。其次，我们应当提示每个智能体在朝目标做出增量式进展的同时，在会话结束时把环境恢复到一种干净的状态。这里的“干净状态”，指的是足以合并进主分支的代码状态：没有重大 bug，代码结构整洁且文档清晰，总体而言，开发者无需先清理一堆无关混乱，就可以直接开始开发新功能。

在内部实验中，我们通过一个两段式方案来解决这些问题：

- **初始化智能体（Initializer agent）：** 第一次智能体会话会使用一个专门的提示，要求模型完成初始环境搭建：创建一个 `init.sh` 脚本、一个用于记录智能体工作日志的 `claude-progress.txt` 文件，以及一个初始 git 提交，用来展示新增了哪些文件。
- **编码智能体（Coding agent）：** 之后的每次会话都要求模型做出增量式进展，并留下结构化更新记录。

这里的关键洞察是：必须让智能体在一个全新的上下文窗口中，也能快速理解当前工作状态。这可以通过 `claude-progress.txt` 文件和 git 历史共同实现。我们设计这些实践的灵感，来自于高效软件工程师每天实际采用的工作方式。

## 环境管理

在更新后的 Claude 4 提示指南中，我们分享了一些多上下文窗口工作流的最佳实践，其中包括一种 harness 结构：为“第一个上下文窗口使用不同的提示”。这个“不同的提示”会要求初始化智能体把未来编码智能体高效工作所需的上下文全部准备好。下面我们进一步深入介绍这种环境中的几个关键组成部分。

### 功能列表

为了解决智能体试图一次性完成整个应用，或过早认定项目完成的问题，我们会提示初始化智能体写出一份完整的功能需求文件，将用户最初的提示展开为更细粒度的功能列表。以 claude.ai 克隆版为例，这意味着要列出 200 多项功能，例如“用户可以打开一个新聊天，输入问题，按下回车，并看到 AI 回复”。这些功能在初始状态下都会被标记为“失败”，这样后续的编码智能体就能清楚知道“完整功能”应当是什么样子。

JSON 条目示例：

```json
{
  "category": "functional",
  "description": "新聊天按钮可以创建新的会话",
  "steps": [
    "进入主界面",
    "点击‘New Chat’按钮",
    "确认已创建新的会话",
    "检查聊天区域是否显示欢迎状态",
    "确认该会话出现在侧边栏中"
  ],
  "passes": false
}
```

我们会提示编码智能体只能通过修改 `passes` 字段的状态来编辑这个文件，并使用措辞非常强硬的指令，例如“绝对不能删除或编辑测试，因为这可能导致功能缺失或存在 bug”。经过一些实验后，我们最终选择使用 JSON 来存储这些内容，因为相比 Markdown 文件，模型不太容易对 JSON 文件进行不恰当的修改或覆盖。

### 增量式进展

在有了上述初始环境脚手架之后，编码智能体的下一版提示便会要求它一次只处理一个功能。这种增量式方法被证明对于解决智能体“同时做太多事”的倾向至关重要。

即便采用增量式开发，模型在完成代码修改后，仍然必须把环境留在一个干净状态。在实验中，我们发现，最有效的做法是要求模型把进展提交到 git，并编写描述清晰的提交信息，同时在进度文件中写下本次工作的摘要。这样模型就可以借助 git 回滚错误代码，并恢复代码库中的可用状态。

这些做法也提升了效率，因为智能体不再需要猜测之前到底发生了什么，也不必花时间重新让基础应用恢复工作。

### 测试

我们观察到的最后一种主要失败模式，是 Claude 倾向于在没有充分测试的情况下把某个功能标记为已完成。如果没有显式提示，Claude 往往会做代码修改，也会进行一些测试，例如编写单元测试，或对开发服务器执行 curl 命令，但它常常无法识别：该功能实际上并没有真正做到端到端可用。

在构建 Web 应用的场景中，一旦明确提示 Claude 使用浏览器自动化工具，并像真实用户那样完成全部测试流程，它通常就能更好地验证功能的端到端效果。

向 Claude 提供这类测试工具后，性能有了显著提升，因为智能体能够识别并修复那些仅从代码本身并不明显的 bug。

不过，一些问题仍然存在，例如 Claude 视觉能力的限制，以及浏览器自动化工具本身的局限，使它难以识别所有类型的 bug。举例来说，Claude 无法通过 Puppeteer MCP 看见浏览器原生 alert 模态框，因此依赖这类模态框的功能通常会更容易出错。

## 快速进入状态

在上述机制全部就位后，每个编码智能体在开始工作时，都会被提示按一系列步骤来快速了解当前状态：

- 运行 `pwd`，确认你正在工作的目录。
- 阅读 git 日志和进度文件，了解最近做了什么。
- 阅读功能列表文件，并选择一个尚未完成、优先级最高的功能进行处理。

如果再让初始化智能体额外写一个 `init.sh` 脚本，用于启动开发服务器，并在实现新功能前先跑一遍基础端到端测试，也会很有帮助。

一个典型会话的开场流程通常如下：读取进度记录 → 读取功能列表 → 查看 git 日志 → 运行 `init.sh`/启动开发服务器 → 通过浏览器自动化执行基础 E2E 检查。

## 未来工作

这些研究记录也提出了一些开放问题：

- 在跨上下文场景中，是单一的通用编码智能体更合适，还是多智能体协作方案（例如专门负责测试/QA/清理的智能体）表现更好？
